Extend packages/types/src/panel.ts:

Add three new sub-schemas before BasePanelWidgetSchema:

LinePointSchema: z.object({ x: z.string().max(60), y: z.number() }) — x is a label or date string, y is numeric.

TimelineEventSchema: z.object({ date: z.string().max(40), label: z.string().max(120), description: z.string().max(240).optional() }) — date is a display string (e.g. "2024-01", "Jan 2024"), label is the event name, description is optional detail.

MapCountrySchema: z.object({ code: z.string().length(2), label: z.string().max(80), value: z.number().optional() }) — code is ISO 3166-1 alpha-2 (e.g. "US", "BR"), label is display name, value is optional numeric.

Add three new variants to BasePanelWidgetSchema's discriminatedUnion options (after the existing 'bar' entry):
- widget_type: z.literal('line'), title: z.string().optional(), points: z.array(LinePointSchema).min(2).max(50)
- widget_type: z.literal('timeline'), title: z.string().optional(), events: z.array(TimelineEventSchema).min(1).max(20)
- widget_type: z.literal('map'), title: z.string().optional(), countries: z.array(MapCountrySchema).min(1).max(50), highlight_color: z.string().optional() (hex string, e.g. "#6366f1")

PanelWidgetSchema spread already uses BasePanelWidgetSchema.options — no change needed there since the new types become part of the base options. However, BasePanelWidgetSchema CANNOT include 'layout' (self-reference constraint noted in the file). All three new types are non-recursive so they belong in BasePanelWidgetSchema.

Extend packages/types/src/ai.ts:

In renderPanelTool.parameters.properties.widget_type.enum, add 'line', 'timeline', 'map' to the array.

Update renderPanelTool.description to add: "line → points[]; timeline → events[]; map → countries[]."

Add three new property entries in renderPanelTool.parameters.properties (after the existing 'widgets' entry):

points property (already exists for scatter — this is a DIFFERENT usage for line): Do NOT add a duplicate 'points' key. Instead, note in the line variant description that it reuses the 'points' property. Actually, look at the existing schema: scatter uses 'points' with fields {concept, consensus, impact}, while line needs {x, y}. These are different shapes. The tool parameters object is a JSON Schema object — property names must be unique. Resolution: name the line data property 'line_points' in the tool parameters (not 'points') so it does not collide with scatter's 'points'. Update LinePointSchema in panel.ts to match: the Zod schema property name does not need to match the tool parameter name — but for clarity, align them. Use 'line_points' as both the Zod schema field name and the tool parameter name.

So in panel.ts: line variant uses line_points: z.array(LinePointSchema).min(2).max(50).

In ai.ts tool parameters, add:
- line_points: array description "Required when widget_type=line. 2–50 data points. Each point: { x: string (label or date), y: number }."
- events: array description "Required when widget_type=timeline. 1–20 events. Each: { date: string, label: string, description?: string }."
- countries: array description "Required when widget_type=map. 1–50 countries. Each: { code: ISO alpha-2 string (e.g. 'US'), label: string, value?: number }."
- highlight_color: string description "Optional hex color for map highlights (e.g. '#6366f1'). Defaults to indigo if omitted."

Also update renderPanelTool.description to mention: "line → line_points[]; timeline → events[]; map → countries[]."
