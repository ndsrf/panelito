## 1. System Classification

**System Type:** Conversational (streaming chat + structured tool-call output via `render_panel`)

**Description:**
Phase 4 extends Project Multiverse's AI integration layer so OpenAI and Gemini can serve as first-class alternatives to Anthropic. The system streams chat tokens to all session participants in real time while simultaneously accumulating tool call arguments for the `render_panel` tool, which drives the live analytics panel. The user is the session creator — a facilitator or researcher who has configured their preferred provider and API key in `/settings`. "Good" means: text tokens flow to chat within 400 ms of first token, the panel updates with a validated `PanelWidget` payload within 2 s of conversation turn end, and switching providers in `/settings` requires no other changes from the creator or participants. The system is provider-unaware downstream of the adapter boundary — route logic, frontend state, and Supabase subscriptions are identical regardless of which provider is active.

**Critical Failure Modes:**
1. **Tool call JSON emitted from partial streaming chunks** — all three SDKs stream function/tool call arguments as deltas. If an adapter yields a `tool_use` event before the full JSON is accumulated, `PanelWidgetSchema.safeParse()` will receive a truncated object, fail validation, and drop the panel update silently. The user sees stale or missing panel data with no error.
2. **System prompt injected into the wrong field** — Anthropic uses a top-level `system` parameter; OpenAI expects `{ role: "system", content: "..." }` as the first message in the array; Gemini uses `config.systemInstruction`. Passing the system string in the wrong position results in it being treated as user content, breaking persona behavior and structured output compliance across all downstream turns.
3. **Provider adapter throws instead of yielding a `done` event** — if the adapter errors mid-stream and throws rather than emitting `{ type: "done" }`, the SSE route has no clean close signal, leaving the client waiting indefinitely. All participants in the session see a frozen panel and no spinner resolution.
4. **`PanelWidgetSchema.safeParse()` gate bypassed or weakened** — if a new adapter emits a `tool_use` event with `input` that skips the Zod validation gate, invalid widget payloads reach Zustand's `panelStore` and corrupt the panel state for all participants for the rest of the session.
5. **Concurrent tool call index mis-tracking (OpenAI)** — OpenAI streaming allows parallel tool calls, each identified by `index`. If the accumulation map does not key by `index`, arguments from separate concurrent calls overwrite each other, producing malformed JSON that fails `JSON.parse()` and drops the panel update.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Collaborative intelligence / group facilitation tooling — digital tools that assist teams or groups in structured deliberation, debate, and decision-making (peers: Mentimeter, Miro AI, Kialo, Pol.is, Real-Time Delphi platforms)

**User Population:** Session creators who are facilitators, researchers, or team leads running live structured group debates. Participants are session guests (peers, teammates, research subjects) who contribute messages and view the analytics panel in real time. The creator is the primary domain expert; participants are the people whose collective thinking the tool is supposed to surface.

**Stakes Level:** High — a live group session is a time-bounded, synchronous event with multiple participants present. A panel failure or AI behavior gap cannot be replayed; the moment is lost. Provider-switching is invisible to participants, so quality degradation under a non-default provider will be attributed to the platform, not the provider.

**Output Consequence:** The analytics panel is the primary deliverable of each AI turn in a session. If a provider produces a stale panel, wrong widget type, or schema-invalid payload, participants lose visibility into the group's synthesized thinking at the exact moment they need it. Facilitators who use the panel to steer the discussion cannot course-correct without it. For asynchronous review or session export, a missing panel state is a permanent gap in the session record.

### What Domain Experts Evaluate Against

**Dimension: Panel timeliness relative to conversation pace**
- Good: The analytics panel updates within 2 seconds of a conversation turn ending — participants can see the synthesized state before the next message arrives.
- Bad: The panel is still loading (spinner visible) when participants have already sent 2-3 follow-up messages; the panel feel disconnected from the conversation.
- Stakes: Critical
- Source: Facilitation practitioner expectation — "AI facilitation tools should instantly visualize everyone's opinion" (Mentimeter, SessionLab State of Facilitation 2025). Live session tools like Miro and Mentimeter set the benchmark for real-time display latency.

**Dimension: Widget type fit for the conversation content**
- Good: When the group is debating trade-offs across multiple dimensions, the panel shows a radar chart; when the group is converging on a ranked list, the panel shows a bento grid of key concept cards; when voting on proportions, a pie chart.
- Bad: Every turn produces the same widget type regardless of conversation content (e.g., always a radar chart), or the widget type contradicts what participants are discussing (scatter plot during a simple Q&A).
- Stakes: High
- Source: Core value proposition of Project Multiverse (REQUIREMENTS PANEL-01); domain knowledge that facilitators use visual artifacts to reflect the group's cognitive state back to them — the wrong artifact type disrupts sense-making.

**Dimension: Analytical tone consistency across providers**
- Good: The Scientific Analyst persona responds with equivalent clinical tone, logical structure, and level of hedging regardless of whether the session creator's active provider is Anthropic, OpenAI, or Gemini.
- Bad: Switching from Anthropic to Gemini causes the AI to become conversational and verbose where it was previously terse and structured, or to stop flagging logical fallacies it previously flagged.
- Stakes: High
- Source: Cross-provider output consistency research (Deepchecks, Cleanlab 2024-2025); practitioner knowledge that persona inconsistency breaks facilitator trust in the tool across sessions.

**Dimension: Schema compliance rate for `render_panel` payloads**
- Good: All three providers produce `render_panel` tool call arguments that pass `PanelWidgetSchema.safeParse()` on the first attempt in ≥ 95% of turns.
- Bad: One provider produces schema-valid payloads reliably while another silently drops panel updates in 20%+ of turns due to structural differences in how it handles nested JSON in tool arguments.
- Stakes: Critical
- Source: AI-05 requirement (REQUIREMENTS.md); established practitioner knowledge that schema non-compliance from cheaper or different-generation models is a primary cross-provider parity failure mode in production structured-output deployments.

**Dimension: Invisible provider switch — no session disruption**
- Good: A facilitator who switches their active provider in `/settings` between sessions sees no behavioral difference from the participants' perspective — same first-token latency envelope, same panel update cadence, same widget fidelity.
- Bad: After a provider switch, participants notice the panel stops updating, response latency jumps from 1s to 5s, or the AI starts producing plain-text responses that do not update the panel at all.
- Stakes: High
- Source: Phase 4 goal (D-14, D-05); facilitator experience standard — tool reliability is binary in a live session context (it works or it doesn't).

### Known Failure Modes in This Domain

**1. Premature convergence induced by AI output framing**
Group deliberation research documents that AI-generated summaries or visualizations that present consensus prematurely cause participants to anchor on the AI's framing rather than surface their actual disagreements. In this product, a radar chart that normalizes all dimensions to similar scores when participants actually hold divergent views can shut down productive debate. This is not a hallucination — the widget data may be structurally valid but semantically misleading.
Source: Shared information bias research (Wikipedia / group decision-making literature); SessionLab State of Facilitation 2025 — "AI cannot differentiate between what is relevant and what is merely noise."

**2. Provider-silent panel failure — stale panel with no error signal**
When `PanelWidgetSchema.safeParse()` fails silently (as designed — AI-05 says to discard without crashing), participants see a frozen panel with no explanation. In a facilitation context, participants interpret a frozen panel as a tool failure and lose confidence in the session. Unlike a crash (which is visible), a silent discard means the facilitator has no signal that the panel is stale. This is specific to multi-provider deployments: Anthropic may reliably produce valid schema payloads while a newly-added Gemini adapter silently fails.

**3. Dominant voice amplification via AI reflection**
If one participant sends longer, more structured messages, the AI's panel summaries may over-weight their framing (shared information bias — established in group decision-making literature). In this product, the analytics widget could reflect one participant's conceptual model rather than the group's, especially under providers that are more sensitive to message length as a signal of importance. Facilitators evaluating AI-assisted group tools specifically watch for AI outputs that flatten the diversity of contributions.

**4. Latency-induced session freeze perception**
In a synchronous group setting, a provider that reliably delivers the first text token in < 400ms but delivers the panel update in > 4000ms creates a perceptual split: participants see the AI responding in chat but see the panel frozen. Facilitators interpret this as the system being "broken." This failure mode is specific to the dual-channel streaming architecture (AI-04) and is most likely to surface with providers that batch tool call arguments rather than streaming them.

### Regulatory / Compliance Context

**GDPR / data residency (relevant for EU deployments):**
All three providers (Anthropic, OpenAI, Google Gemini) process data in US infrastructure by default. When session content includes personally identifiable information about EU residents (e.g., participant names, discussion content about identifiable individuals), GDPR Article 46 transfer obligations apply. The BYOK model means the creator's API key routes their session data to the provider they select — the platform does not control data residency. This is a creator responsibility, not a platform obligation in v1, but the `/settings` UI should surface a brief disclosure that the creator's chosen provider processes session data according to that provider's terms.
Source: EDPB 2025 LLM guidance; regolo.ai / Janus Compliance GDPR-AI analysis 2026.

**No direct HIPAA or FCA applicability identified:** Project Multiverse v1 targets strategy, debate, and red team use cases — not healthcare or financial advisory. HIPAA and FCA are not directly applicable unless a specific deployment context introduces regulated data. This is a deployment-context risk, not a platform architecture risk.

**Consent for AI participation in group sessions:**
In jurisdictions with AI transparency requirements (EU AI Act, effective August 2026), participants in an AI-assisted group session should be informed that an AI persona is present and contributing. The existing UI (PERSONA-03 — each AI response bubble is labeled with the persona's name and icon) satisfies transparency requirements for the interactive chat component. No additional regulatory action required for v1 in this respect.

**API key security:**
BYOK keys are stored encrypted in Supabase user_metadata (AI-01, AI-02) and never transmitted to the browser. This pattern is compliant with standard API key handling expectations across all three providers' terms of service.

## 2. Framework Decision

**Selected Framework:** Raw SDK Adapters — `openai` npm SDK (TypeScript) + `@google/genai` v2 SDK (TypeScript)

**Version:** openai ^6.x (latest stable, currently 6.44.0), @google/genai ^2.x (v2 SDK — currently 2.8.0 — NOT @google/generative-ai)

**Rationale:**
Phase 4 is an adapter implementation task behind an already-designed `AIProvider` interface — not a new AI system requiring orchestration, RAG, or multi-agent coordination. The existing codebase has deliberately excluded every abstraction layer not earning its weight (no LangChain, no tRPC, no Redux per CLAUDE.md). Adding a framework beneath the team-owned `AIStreamEvent` contract and `contentBlock` accumulation pattern would mean mapping a third party's event model to types the team already controls, adding upgrade surface and debugging indirection with zero compensating benefit.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain.js | Provides cross-provider streaming unification, but its event model conflicts with the existing `AIStreamEvent` contract — the mapping cost exceeds the cost of writing the adapters directly; also violates CLAUDE.md's explicit exclusion of abstraction overhead |
| LangGraph (TypeScript) | Stateful graph model adds significant complexity for a pure adapter task; no benefit over direct SDK calls when no branching workflow or checkpointing is needed |

**Vendor Lock-In Accepted:** No — the `AIProvider` interface is the team's own abstraction; any adapter can be swapped or removed without touching route logic

### Entry Point Pattern

#### OpenAI adapter — streaming + tool call accumulation
```typescript
import OpenAI from 'openai'
import type { ChatCompletionMessageParam, ChatCompletionTool } from 'openai/resources/chat/completions'
import type { ProviderMessage, ProviderTool, AIStreamEvent } from '@multiverse/types'
import { PanelWidgetSchema } from '@multiverse/types'

export class OpenAIAdapter implements AIProvider {
  constructor(private readonly apiKey: string) {}

  async *stream(
    messages: ProviderMessage[],
    tools: ProviderTool[],
    options: { model: string; maxTokens: number; system?: string }
  ): AsyncIterable<AIStreamEvent> {
    const client = new OpenAI({ apiKey: this.apiKey })

    // Convert ProviderMessage[] to OpenAI format.
    // System prompt goes as the FIRST message with role "system" — not a top-level param.
    const oaiMessages: ChatCompletionMessageParam[] = []
    if (options.system) {
      oaiMessages.push({ role: 'system', content: options.system })
    }
    for (const m of messages) {
      // ProviderMessage.role can be 'system' | 'user' | 'assistant'
      // Route only sends 'user' / 'assistant' in messages[]; system comes via options.system
      oaiMessages.push({ role: m.role as 'user' | 'assistant', content: m.content })
    }

    // Convert ProviderTool[] to OpenAI ChatCompletionTool format
    const oaiTools: ChatCompletionTool[] = tools.map((t) => ({
      type: 'function',
      function: {
        name: t.name,
        description: t.description,
        parameters: t.parameters,
      },
    }))

    const stream = await client.chat.completions.create({
      model: options.model,
      max_tokens: options.maxTokens,
      messages: oaiMessages,
      tools: oaiTools,
      stream: true,
    })

    // Accumulate tool call arguments by index — OpenAI streams partial JSON strings.
    // Keys are the tool call index (supports concurrent parallel tool calls).
    const toolCallAccumulator: Record<
      number,
      { id: string; name: string; arguments: string }
    > = {}

    for await (const chunk of stream) {
      const delta = chunk.choices[0]?.delta
      if (!delta) continue

      // Yield text deltas immediately
      if (delta.content) {
        yield { type: 'text_delta', text: delta.content }
      }

      // Accumulate tool call argument fragments
      for (const tc of delta.tool_calls ?? []) {
        const idx = tc.index
        if (!toolCallAccumulator[idx]) {
          // First delta for this index carries id and name
          toolCallAccumulator[idx] = {
            id: tc.id ?? '',
            name: tc.function?.name ?? '',
            arguments: '',
          }
        }
        // Subsequent deltas only carry arguments fragments
        toolCallAccumulator[idx].arguments += tc.function?.arguments ?? ''
      }

      // When finish_reason is 'tool_calls', all argument strings are complete
      if (chunk.choices[0]?.finish_reason === 'tool_calls') {
        for (const tc of Object.values(toolCallAccumulator)) {
          let parsed: unknown
          try {
            parsed = JSON.parse(tc.arguments)
          } catch {
            // Malformed JSON from provider — drop this tool call
            continue
          }
          yield { type: 'tool_use', name: tc.name, input: parsed }
        }
      }
    }

    yield { type: 'done' }
  }
}
```

#### Gemini adapter — generateContentStream + function call accumulation
```typescript
import { GoogleGenAI, FunctionCallingConfigMode } from '@google/genai'
import type { FunctionDeclaration } from '@google/genai'
import type { ProviderMessage, ProviderTool, AIStreamEvent } from '@multiverse/types'
import { PanelWidgetSchema } from '@multiverse/types'

export class GeminiAdapter implements AIProvider {
  constructor(private readonly apiKey: string) {}

  async *stream(
    messages: ProviderMessage[],
    tools: ProviderTool[],
    options: { model: string; maxTokens: number; system?: string }
  ): AsyncIterable<AIStreamEvent> {
    const ai = new GoogleGenAI({ apiKey: this.apiKey })

    // Convert ProviderTool[] to Gemini FunctionDeclaration format.
    // Gemini uses parametersJsonSchema (standard JSON Schema object).
    const functionDeclarations: FunctionDeclaration[] = tools.map((t) => ({
      name: t.name,
      description: t.description,
      parametersJsonSchema: t.parameters,
    }))

    // Convert ProviderMessage[] to Gemini contents format.
    // Gemini does not support 'system' role in contents — use config.systemInstruction instead.
    const contents = messages
      .filter((m) => m.role !== 'system')
      .map((m) => ({
        role: m.role === 'assistant' ? 'model' : 'user',
        parts: [{ text: m.content }],
      }))

    const responseStream = await ai.models.generateContentStream({
      model: options.model,
      contents,
      config: {
        maxOutputTokens: options.maxTokens,
        // System prompt goes here — NOT in contents
        ...(options.system ? { systemInstruction: options.system } : {}),
        tools: functionDeclarations.length > 0
          ? [{ functionDeclarations }]
          : undefined,
      },
    })

    // Gemini emits function calls as complete objects in chunk.functionCalls,
    // available after accumulating the full response. During streaming, text
    // arrives in chunk.text; function call parts arrive in the final chunks.
    // Accumulate function calls across all chunks before emitting tool_use.
    let accumulatedFunctionCalls: Array<{ name: string; args: unknown }> = []

    for await (const chunk of responseStream) {
      // Text deltas — yield immediately
      if (chunk.text) {
        yield { type: 'text_delta', text: chunk.text }
      }

      // Function calls — Gemini surfaces them via chunk.functionCalls accessor.
      // This is populated only when the chunk contains a functionCall Part.
      if (chunk.functionCalls && chunk.functionCalls.length > 0) {
        for (const fc of chunk.functionCalls) {
          accumulatedFunctionCalls.push({ name: fc.name ?? '', args: fc.args ?? {} })
        }
      }
    }

    // Emit all accumulated function calls after stream completes
    for (const fc of accumulatedFunctionCalls) {
      yield { type: 'tool_use', name: fc.name, input: fc.args }
    }

    yield { type: 'done' }
  }
}
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `stream: true` + `finish_reason: "tool_calls"` (OpenAI) | Chat Completions streaming flag; `finish_reason` signals all tool call argument strings are complete and safe to `JSON.parse()` | Use to know when to flush the `toolCallAccumulator` map and emit `tool_use` events — never parse argument fragments mid-stream |
| `delta.tool_calls[].index` (OpenAI) | Integer key identifying which concurrent tool call a delta fragment belongs to — id and name only appear on the first delta per index | Use as the accumulation map key to avoid merging arguments from separate parallel tool calls |
| `generateContentStream()` (Gemini) | Returns `Promise<AsyncGenerator<GenerateContentResponse>>` — the async generator yields `GenerateContentResponse` chunks with `.text` and `.functionCalls` accessors | Use for all Gemini streaming; `chunk.text` is undefined between function call chunks, so guard before yielding |
| `chunk.functionCalls` accessor (Gemini) | Convenience accessor on `GenerateContentResponse` that returns `FunctionCall[]` when the chunk contains a `functionCall` Part; undefined otherwise | Use to detect and accumulate function calls during streaming — treat as complete objects (Gemini does not stream partial function args like OpenAI does) |
| `config.systemInstruction` (Gemini) | Top-level config field for system prompt — completely separate from `contents` | Pass the shared system prompt here; never put it in `contents` with role `"system"` (Gemini rejects that role value) |

### Common Pitfalls

1. **Parsing OpenAI `function.arguments` before `finish_reason === "tool_calls"`** — `function.arguments` in streaming chunks is a raw string fragment (e.g., `'{"widget_'`). Calling `JSON.parse()` on a fragment throws a `SyntaxError` that silently drops the panel update if uncaught, or crashes the generator if uncaught. Only call `JSON.parse()` after `finish_reason` is `"tool_calls"`.

2. **Using `role: "system"` in Gemini `contents`** — The `@google/genai` SDK will throw or ignore entries with `role: "system"` in the `contents` array. The system prompt must go in `config.systemInstruction`. Filtering out `ProviderMessage` entries with `role === "system"` before building `contents` is mandatory.

3. **Missing `index` key in OpenAI tool call accumulator** — When `parallel_tool_calls` is enabled (default), multiple tool call deltas can arrive interleaved in the same stream. If you concatenate `function.arguments` into a single string instead of keying by `index`, you produce invalid JSON. Always key the accumulator by `tc.index`.

4. **Treating Gemini streaming function calls as partial (like OpenAI)** — Unlike OpenAI, Gemini does not stream partial function call arguments. `chunk.functionCalls` contains complete, already-parsed objects. Attempting to accumulate and re-parse them as strings will double-encode the JSON.

5. **Not converting Gemini `role: "assistant"` to `role: "model"`** — Gemini's `contents` format uses `"model"` (not `"assistant"`) for AI turns. Passing `role: "assistant"` will cause a 400 validation error from the Gemini API.

**Model Configuration:**

| Gemini | compression / categorization | `gemini-2.5-flash` | Single model covers all Gemini tasks at low cost |

**gpt-4o and gpt-4o-mini are no longer available in the OpenAI API as of February 2026. Do not use these model IDs.** Use `gpt-5.4` and `gpt-5.4-mini` respectively.

**gemini-2.0-flash was shut down June 1, 2026.** Use `gemini-2.5-flash` as the replacement for all Gemini tasks.

Recommended parameters for `analysis` (primary streaming task):
- `maxTokens`: 2048 (cap streaming responses — never leave unbounded in production)
- `temperature`: not set (use provider defaults for conversational tasks; Gemini does not expose temperature in the same way OpenAI does)

**Core Pattern:**

All three adapters implement the same `contentBlock` accumulation invariant established in Phase 2:

> Tool call / function call JSON is emitted as a single complete `{ type: "tool_use", name, input }` event ONLY after the full input object is available — never from partial streaming chunks.

**OpenAI:** Accumulate `delta.tool_calls[].function.arguments` strings keyed by `delta.tool_calls[].index`. Flush to `tool_use` events only when `finish_reason === "tool_calls"`. Then `JSON.parse()` each accumulated string.

**Gemini:** `chunk.functionCalls` returns complete, already-parsed `FunctionCall` objects (not partial strings). Accumulate across chunks, emit after stream ends. No `JSON.parse()` needed.

**Anthropic (existing):** Uses the SDK's `contentBlock` event which fires after `content_block_stop` with fully-assembled `block.input`. Pattern is identical in effect.

After emitting all `tool_use` events (which the route passes to `PanelWidgetSchema.safeParse()`), yield `{ type: "done" }` unconditionally — even if no tool calls occurred.

**Tool Use:**

The `renderPanelTool` definition must be converted from `ProviderTool` format to each SDK's native format inside the adapter. The route passes `ProviderTool[]` and never imports SDK-specific types.

OpenAI format (in `OpenAIAdapter`):
```typescript
// ProviderTool -> ChatCompletionTool
const oaiTools: ChatCompletionTool[] = tools.map((t) => ({
  type: 'function',
  function: {
    name: t.name,
    description: t.description,
    parameters: t.parameters, // JSON Schema — passed through as-is
  },
}))
```

Gemini format (in `GeminiAdapter`):
```typescript
// ProviderTool -> FunctionDeclaration
const functionDeclarations: FunctionDeclaration[] = tools.map((t) => ({
  name: t.name,
  description: t.description,
  parametersJsonSchema: t.parameters, // JSON Schema — passed through as-is
}))

// Passed inside config.tools as a Tool object:
config: {
  tools: [{ functionDeclarations }]
}
```

The route constructs `ProviderTool[]` from the existing `renderPanelTool` definition (refactored to `ProviderTool` shape in Phase 4) and passes it to `adapter.stream()`. Each adapter converts internally without touching route logic.

**State Management:**
Provider selection is a global setting stored in Supabase user_metadata (`active_provider: "anthropic" | "openai" | "gemini"`). Keys are namespaced (`anthropic_api_key`, `openai_api_key`, `gemini_api_key`). No in-memory state beyond what each request assembles from user_metadata.

**Context Window Strategy:**
Same sliding-window + compression strategy as Phase 2 (D-03, D-08): last 8 raw messages + compressed summary prefix. All adapters receive `ProviderMessage[]` already assembled by the route — adapters do not manage context window.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### 4b.1 Structured Outputs with Zod

The `PanelWidgetSchema` (defined in `packages/types/src/panel.ts`) is the validation gate between raw tool call input and the Zustand `panelStore`. All three adapters must produce `input` objects that pass this gate. The validation happens in the route, not the adapter — but the adapter must ensure `input` is a fully-parsed JavaScript object (not a string) before emitting `tool_use`.

Validation pattern in the route (unchanged from Phase 2, must stay green for all providers):

```typescript
// apps/api/src/routes/ai.ts (route layer — not in adapters)
for await (const event of adapter.stream(messages, tools, options)) {
  if (event.type === 'tool_use' && event.name === 'render_panel') {
    // Gate: reject malformed payloads before touching panelStore
    const result = PanelWidgetSchema.safeParse(event.input)
    if (!result.success) {
      // Log validation errors for debugging — do not surface to user
      console.error('[render_panel] schema validation failed', result.error.flatten())
      // Drop this update — do not crash the stream
      continue
    }
    // Valid payload — broadcast to Supabase Realtime channel
    await supabase.channel(sessionId).send({
      type: 'broadcast',
      event: 'panel_update',
      payload: result.data,
    })
  }

  if (event.type === 'text_delta') {
    // SSE: send text delta to client
    res.write(`data: ${JSON.stringify(event)}\n\n`)
  }

  if (event.type === 'done') {
    res.write(`data: ${JSON.stringify({ type: 'done' })}\n\n`)
    res.end()
  }
}
```

Retry strategy for tool call validation failures: **do not retry at the adapter level**. A failed `safeParse` means the model produced an invalid schema — retrying the same call with the same prompt will likely produce the same error. Log the failure with `result.error.flatten()`, drop the panel update for that turn, and let the stream continue. If validation failures exceed a threshold (e.g., 3 consecutive turns), surface a degraded-mode indicator in the panel (Phase 5 concern).

### 4b.2 Async-First Design

Both SDKs expose streaming as async iterables. The `for await...of` loop is the correct consumption pattern.

**OpenAI:** `client.chat.completions.create({ stream: true })` returns a `Stream<ChatCompletionChunk>` which implements `AsyncIterable<ChatCompletionChunk>`. Iterate with `for await (const chunk of stream)`.

**Gemini:** `ai.models.generateContentStream(params)` returns `Promise<AsyncGenerator<GenerateContentResponse>>`. The `await` resolves the generator, then iterate with `for await (const chunk of responseStream)`.

**The one common mistake** — awaiting the full response instead of iterating:

```typescript
// WRONG — defeats streaming, buffers entire response in memory,
// and first-token latency equals total generation time
const response = await ai.models.generateContent(params)
for (const chunk of response.candidates ?? []) { ... }

// CORRECT — yields chunks as they arrive
const stream = await ai.models.generateContentStream(params)
for await (const chunk of stream) { ... }
```

For OpenAI, the equivalent mistake is calling `client.chat.completions.create()` without `stream: true` and awaiting the full `ChatCompletion` object.

**Stream vs. await rule for this codebase:**
- Text deltas: yield immediately from within the `for await` loop — do not buffer.
- Tool call inputs: await full accumulation (across the entire stream) before emitting `tool_use`. This is the `contentBlock` invariant from Phase 2.
- Never use `await stream.finalMessage()` (OpenAI SDK helper) as it buffers the full response and breaks first-token latency guarantees.

### 4b.3 Prompt Engineering Discipline

**System vs. user prompt separation:** The system prompt defines persona, output format requirements, and the `render_panel` calling contract. It must never appear in the `messages` array passed to the adapter — it comes via `options.system`. Each adapter places it in the correct SDK field:

| Provider | System prompt field |
|----------|-------------------|
| Anthropic | Top-level `system` parameter in `client.messages.stream()` |
| OpenAI | First element of `messages` array: `{ role: "system", content: options.system }` |
| Gemini | `config.systemInstruction` in `generateContentStream()` |

**Provider-neutral system prompt authoring:** Phase 4 uses one shared system prompt for all providers (D-16). Do not mention model names, provider names, or Claude-specific behaviors. Prompt text must produce valid `render_panel` tool calls identically from all three providers. Reference the tool by name only (`render_panel`) — do not reference Anthropic tool use syntax.

**Few-shot examples:** For `render_panel`, include one inline few-shot example in the system prompt showing a well-formed tool call result per widget type. This dramatically improves schema compliance across providers, especially for Gemini which is stricter about schema adherence than Anthropic.

**`max_tokens` / `maxOutputTokens`:** Always set explicitly. The default is unbounded on some providers. `2048` is appropriate for analysis tasks. Compression and categorization tasks can use `512`. Never leave unbounded in production — unbounded responses increase cost linearly and can time out SSE connections.

### 4b.4 Context Window Management

Handled upstream by the route layer (Phase 2 sliding-window pattern). Adapters receive a pre-assembled `ProviderMessage[]` and convert to SDK-native format. Each adapter is responsible for placing the `system` string in the correct SDK field (Anthropic: `system` param; OpenAI: `messages[0].role="system"`; Gemini: `systemInstruction`).

Context window sizes for reference (do not need to enforce in adapters — route handles this):
- `claude-sonnet-4-6`: 200k tokens
- `gpt-5.4`: 128k tokens
- `gemini-2.5-flash`: 1M tokens

The sliding-window strategy (8 raw messages + summary prefix) keeps all providers well within limits for typical sessions.

### 4b.5 Cost and Latency Budget

Model pricing as of June 2026 (verify at provider dashboards before locking budget):

**Task-to-model routing rationale:**

## 5. Evaluation Strategy

### Dimensions

The primary evaluation concern for Phase 4 is **cross-provider parity**: every dimension below is measured per provider (Anthropic, OpenAI, Gemini) and compared. A passing score on Anthropic alone is not a pass.

| Dimension | Rubric | Measurement | Priority |
|-----------|--------|-------------|----------|
| **Schema compliance rate** | PASS: `PanelWidgetSchema.safeParse(event.input)` returns `{ success: true }` for the `render_panel` tool_use event; all required fields present per widget type (e.g., `bento` has `cards[]` with ≥ 1 item; `radar` has `axes[]` with ≥ 3 items); no extra fields that violate the discriminated union. FAIL: `safeParse` returns `{ success: false }` — payload truncated, missing `widget_type`, wrong array key name, or numeric field out of range (e.g., `value: 150` on a 0-100 axis). Target: ≥ 95% pass rate per provider. | Code — Vitest: run each reference input through all three adapters, call `PanelWidgetSchema.safeParse()` on emitted `tool_use` events, count pass/fail per provider | Critical |
| **Streaming event contract compliance** | PASS: Each adapter yields exactly the `AIStreamEvent` union members in the correct sequence — one or more `{ type: "text_delta", text: string }` events (non-empty text), then zero or one `{ type: "tool_use", name: "render_panel", input: object }` event (never mid-stream), then exactly one `{ type: "done" }` event as the final yield. FAIL: Adapter yields `tool_use` before the stream ends; `done` event never emitted (stream hangs); `text_delta` emitted with empty string; more than one `done` event; adapter throws instead of yielding `done`. | Code — Vitest: iterate adapter output with `for await`, assert event sequence shape using a state machine (text_delta* → tool_use? → done); assert `done` is always the last event; assert no throws propagate out of the generator | Critical |
| **Time-to-first-token (TTFT) parity** | PASS: Time from `adapter.stream()` invocation to first `text_delta` event is < 400 ms at P50, < 800 ms at P95, for all three providers. FAIL: Any provider has P50 TTFT > 400 ms or P95 TTFT > 800 ms across reference inputs. Providers must be within 2× of each other at P50 — a provider that consistently takes 1200 ms when Anthropic takes 300 ms is a parity failure even if it is under the absolute cap. | Code — Vitest: record `Date.now()` at stream start and on first `text_delta` event; assert thresholds; run 5 times per input per provider and record p50/p95 | High |
| **Widget type appropriateness** | PASS: Given a conversation excerpt in which the group is performing trade-off analysis across 4+ named dimensions, the provider selects `radar`; for a proportional breakdown question (e.g., "what fraction of our risk is X vs Y?"), the provider selects `pie`; for ranked key concept identification, `bento`; for consensus/impact mapping, `scatter`. FAIL: Provider always returns the same `widget_type` regardless of conversation content (rigid output). Provider returns `radar` for a simple 2-option proportional question. Provider returns `scatter` for a Q&A with no impact/consensus structure. | LLM Judge — send the conversation excerpt + chosen `widget_type` to an LLM with a rubric that checks whether the widget type is appropriate for the discourse structure (not just schema-valid). Calibrate against 5 human-reviewed examples before trusting the judge score. Pass threshold: judge score ≥ 3 on a 1-5 scale | High |
| **Analytical tone consistency across providers** | PASS: Given the same 8-message conversation window (reference input), all three providers produce a chat response in which the Scientific Analyst persona: (a) uses structured or numbered reasoning, not casual conversational prose; (b) hedges factual claims with appropriate epistemic markers ("the evidence suggests", "pending confirmation"); (c) flags at least one logical inconsistency or gap if one is present in the conversation. FAIL: One provider produces a response that is conversational and validating without structure; omits flagging an obvious logical gap that Anthropic's response correctly identified; or uses first-person enthusiasm ("Great point!") inconsistent with the clinical persona. | LLM Judge — score each provider's chat response against a 5-point rubric: 5 = structured, hedged, flags fallacies; 3 = mostly structured but misses one element; 1 = conversational, no structure, no fallacy detection. Calibrate against 5 human-reviewed examples. Target: all three providers score ≥ 3; no provider scores more than 1.5 points below the best-scoring provider for the same input | Medium |
| **Provider switch transparency** | PASS: A session run with Anthropic and a session run with OpenAI on the same 15-example reference dataset show no observable difference in panel update cadence (first panel update arrives within 500 ms of each other at median), widget type selection agreement rate ≥ 80% across inputs, and TTFT difference < 200 ms at P50. FAIL: After switching from Anthropic to OpenAI or Gemini, the panel stops updating in > 5% of reference inputs where Anthropic produced a valid panel update, or TTFT diverges by more than 2× at P50. | Human — developer review of side-by-side session logs (one run per provider on the same conversation thread); compare panel update count, widget type distribution, and TTFT per turn. Run once per provider pair before shipping | Medium |

## 6. Guardrails

### Online (Real-Time)

Online guardrails run on every request. Each one adds latency — only critical failure modes that are catastrophic for a live session are guarded here. All three map directly to critical failure modes from Section 1.

| Guardrail | Trigger | Intervention | Maps to Failure Mode |
|-----------|---------|--------------|----------------------|
| **Schema validation gate** | Every `tool_use` event where `event.name === "render_panel"` — fires on every adapter call, all providers | `PanelWidgetSchema.safeParse(event.input)` already implemented in the route (AI-05). Confirm it is called for OpenAI and Gemini adapter output — not just AnthropicAdapter. On `success: false`: silent drop (current behavior) + `console.error('[render_panel] schema validation failed', result.error.flatten())` + increment a counter for the offline flywheel. No retry. No user-visible error. | Failure mode 1 (partial JSON), Failure mode 4 (gate bypassed) |
| **Stream done-event timeout** | SSE connection open for > 30 seconds without a `{ type: "done" }` event being emitted by the adapter | Close the SSE connection server-side: emit `data: {"type":"error","message":"AI response timed out"}` then `res.end()`. The client treats this as a stream error and re-enables the chat input. The session continues — participants are not frozen. 30 s is chosen because the longest expected generation (2048 tokens at typical streaming rates) completes well under 20 s for all three providers. | Failure mode 3 (adapter throws instead of done) |
| **API key upfront validation** | Creator saves a provider API key in `/settings` | Extend the existing `verifyApiKey()` in `apps/api/src/routes/keys.ts` to accept a `provider` parameter. For each provider: (a) Anthropic: existing handshake; (b) OpenAI: `client.models.list()` with the key and expect 200; (c) Gemini: `ai.models.list()` with the key and expect a non-error response. On failure: block the save, return a specific error message per provider ("OpenAI API key invalid — check key prefix 'sk-...'"). Only persist keys that pass verification. | Failure mode 2 (system prompt in wrong field — prevented by correct adapter code; this guardrail catches keys that would fail at runtime before any session starts) |

### Offline (Flywheel)

Offline metrics run on sampled batches and feed the improvement loop — no session latency impact.

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|-----------------------|
| **Schema pass rate per provider** | Weekly batch: collect all `[render_panel] schema validation failed` log entries from the past 7 days, grouped by `active_provider` from the request context. Calculate pass rate = (total tool_use events − failed events) / total tool_use events per provider. | Alert if pass rate drops below 95% for any provider. Investigate: pull the raw `result.error.flatten()` payloads for the failing provider; identify which schema field is failing (most likely: wrong array key name — `items[]` instead of `cards[]`, or out-of-range numeric values on Gemini). Fix: update the system prompt few-shot example for that widget type or tighten the Promptfoo regression test for that case. |
| **TTFT p95 per provider** | Daily aggregate: record TTFT (ms from request start to first `text_delta` event) for every AI invocation, tagged with provider and model. Compute p50 and p95 per provider per day. | Alert if any provider's p95 TTFT exceeds 800 ms on any given day. Investigate: check for provider-side service degradation (compare against provider status pages). If sustained > 3 days: surface a latency warning in `/settings` next to the provider selector (Phase 5 scope), or consider surfacing a degraded-mode indicator in the session UI. |
| **Tone divergence score** | Monthly sample: pull 20 sessions where the creator switched providers between sessions (identified by `active_provider` changes in user_metadata audit log). For each session pair on the same topic, run the LLM judge tone rubric (see Section 5 dimension) on the first 5 AI responses per provider. Compute mean score per provider and flag divergence where any provider scores more than 1.5 points below the best provider on the same input. | Flag for developer review if divergence > 1.5 for any provider pair on more than 20% of sampled sessions. Review: check system prompt for provider-specific leakage; run the Promptfoo tone eval against the flagged provider; update the shared system prompt if the persona is not being honored. |

## 7. Production Monitoring

**Tracing Tool: Arize Phoenix** (open-source, self-hostable at `http://localhost:6006` in development; deploy as a sidecar in production). Selected because: (a) no LangChain/LangGraph in this stack, ruling out LangSmith; (b) OpenTelemetry-based — instruments any TypeScript HTTP server without framework coupling; (c) strong trace visualization for streaming responses and tool call events; (d) no platform account required for self-hosted deployment.

Instrument the `/invoke` SSE route and each adapter's `stream()` method. Each AI invocation becomes one trace with spans for: (1) adapter selection (provider lookup from user_metadata), (2) stream open, (3) first `text_delta`, (4) `tool_use` event emission, (5) `PanelWidgetSchema.safeParse()` result, (6) `done` event.

**Key Metrics to Track:**

| Metric | Target | How Collected |
|--------|--------|---------------|
| `render_panel` schema pass rate per provider | ≥ 95% per provider | Count `safeParse` success/failure per provider tag in Phoenix spans; expose as a ratio |
| TTFT p50 per provider | < 400 ms | Span duration from stream open to first `text_delta` event; aggregate p50 per provider daily |
| TTFT p95 per provider | < 800 ms | Same span; aggregate p95 per provider daily |
| SSE stream error rate | < 1% of all AI invocations | Count invocations where the done-event timeout guardrail fires (error SSE sent) vs. total invocations |
| API key validation failure rate per provider | < 10% of save attempts per provider | Count `verifyApiKey()` failures per provider in the keys route; alert if > 10% on any day — may indicate a provider service outage or a breaking change in the provider's key format |

**Alert Thresholds:**

| Condition | Threshold | Response |
|-----------|-----------|----------|
| Schema pass rate drops below 95% for any single provider | Sustained ≥ 2 consecutive days | Investigate provider adapter output; run Promptfoo regression suite against that provider; check if provider updated their function-calling behavior |
| TTFT p95 exceeds 800 ms for any provider | Sustained ≥ 3 consecutive days | Check provider status page; consider surfacing a latency warning in `/settings` (Phase 5 scope) |
| SSE stream error rate exceeds 1% | Any single day | Check if the done-event timeout (30 s) is being hit; review adapter error handling; confirm all three adapters yield `done` unconditionally in the finally block |
| API key validation failure rate exceeds 10% for any provider | Any single day | Check provider API status; the failure rate jump indicates provider-side authentication infrastructure issues, not user error |
| Tone divergence score exceeds 1.5 for any provider pair | Monthly sample review | Review system prompt for provider-specific leakage; re-run Promptfoo LLM judge eval for flagged provider |

**Smart Sampling Strategy:**

Weight trace collection toward events most likely to surface adapter-specific failures — avoid uniform sampling which dilutes signal with uneventful turns.

- **Weight 3× toward `tool_use` events** — every turn that emits a `render_panel` tool call is higher signal than text-only turns. These are the turns where schema compliance failures, JSON accumulation bugs, and partial-chunk errors will surface.
- **Weight 3× toward the first session after a creator switches active provider** — capture the full trace of the first AI invocation under a new provider setting. This surfaces system prompt field errors, key validation gaps, and adapter initialization failures at the most likely moment they occur.
- **Weight 2× toward any session that contains an error event in the SSE stream** — turns where the timeout guardrail fired or where the adapter emitted an error SSE are the most informative for debugging. Review 100% of these traces regardless of volume.
- **Baseline (1×):** sample 10% of all other AI invocations uniformly across providers. This maintains coverage for latency drift and schema degradation trends without overwhelming trace storage.
