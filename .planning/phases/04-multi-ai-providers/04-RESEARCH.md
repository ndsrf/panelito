## Summary

Phase 4 is an adapter implementation task behind an already-designed `AIProvider` interface. The codebase has a clean Anthropic-only `AnthropicAdapter` in `apps/api/src/lib/ai-provider.ts` that needs to be refactored into a provider-agnostic interface with two new adapter siblings (`OpenAIAdapter`, `GeminiAdapter`). The AI-SPEC.md generated by `gsd-ai-researcher` before this research phase already contains detailed, validated implementation patterns for both new adapters — including the critical `contentBlock` accumulation invariant, system prompt placement per provider, and the `tool_calls index` tracking pattern for OpenAI concurrent tool calls.

The primary complexity is not algorithmic — it is structural: the existing codebase has Anthropic types (`MessageParam`, `Anthropic.Tool`) hardwired into the `AIProvider` interface signature, into `assemblePromptArray()` return type, into the `/invoke` route, and into the `creator_settings` DB schema (one key column, Anthropic-only). Phase 4 must thread the refactoring across all these layers without breaking existing Phase 2 behavior. The `assemblePromptArray()` function uses Anthropic's `cache_control` extension — this must be isolated to the `AnthropicAdapter` and not bleed into the provider-agnostic `ProviderMessage` type.

The `/settings` page and `creator_settings` DB schema need a Supabase migration to add three new encrypted key columns and an `active_provider` column, plus a UI overhaul from a single-provider card to a three-provider selector. The `keys.ts` route and `settings.ts` route need extension for multi-provider key management.

**Primary recommendation:** Follow the AI-SPEC.md patterns exactly. The accumulation pattern, system prompt placement, and Gemini role mapping are all pre-validated. The main planning risk is ordering refactoring steps — the interface must be changed before adapters are written, and the DB migration must land before the routes can reference new columns.

### Pattern 1: contentBlock Accumulation Invariant (ALL adapters must implement)

**What:** Tool call JSON is emitted as a single complete `{ type: "tool_use", name, input }` event ONLY after the full input is available. Never from partial streaming chunks.

**When to use:** Every adapter's `stream()` generator, before yielding `tool_use` events.

**OpenAI pattern** — accumulate by `index`, flush on `finish_reason === "tool_calls"`:
```typescript
// Source: 04-AI-SPEC.md §3 — Framework Quick Reference
const toolCallAccumulator: Record<number, { id: string; name: string; arguments: string }> = {}

for await (const chunk of stream) {
  const delta = chunk.choices[0]?.delta
  if (!delta) continue

  if (delta.content) {
    yield { type: 'text_delta', text: delta.content }
  }

  for (const tc of delta.tool_calls ?? []) {
    const idx = tc.index
    if (!toolCallAccumulator[idx]) {
      toolCallAccumulator[idx] = { id: tc.id ?? '', name: tc.function?.name ?? '', arguments: '' }
    }
    toolCallAccumulator[idx].arguments += tc.function?.arguments ?? ''
  }

  if (chunk.choices[0]?.finish_reason === 'tool_calls') {
    for (const tc of Object.values(toolCallAccumulator)) {
      let parsed: unknown
      try { parsed = JSON.parse(tc.arguments) } catch { continue }
      yield { type: 'tool_use', name: tc.name, input: parsed }
    }
  }
}
yield { type: 'done' }
```

**Gemini pattern** — `chunk.functionCalls` returns complete objects (not partial strings); accumulate across chunks, emit after loop:
```typescript
// Source: 04-AI-SPEC.md §3 — Framework Quick Reference
let accumulatedFunctionCalls: Array<{ name: string; args: unknown }> = []

for await (const chunk of responseStream) {
  if (chunk.text) {
    yield { type: 'text_delta', text: chunk.text }
  }
  if (chunk.functionCalls && chunk.functionCalls.length > 0) {
    for (const fc of chunk.functionCalls) {
      accumulatedFunctionCalls.push({ name: fc.name ?? '', args: fc.args ?? {} })
    }
  }
}

for (const fc of accumulatedFunctionCalls) {
  yield { type: 'tool_use', name: fc.name, input: fc.args }
}
yield { type: 'done' }
```

### Pattern 5: Per-Provider Key Verification

The current `verifyApiKey()` in `apps/api/src/lib/anthropic.ts` does a minimal handshake with `client.messages.create({ max_tokens: 1 })`. The same pattern exists for each provider:

**OpenAI:** `client.models.list()` — returns 200 with a valid key; throws `AuthenticationError` on invalid key.
**Gemini:** `ai.models.list()` — returns model list with a valid key; throws on auth failure.

```typescript
// Extend apps/api/src/lib/anthropic.ts → rename or add multi-provider verify
// Or create a new apps/api/src/lib/verify-key.ts

export async function verifyOpenAIKey(key: string): Promise<VerifyResult> {
  const client = new OpenAI({ apiKey: key })
  try {
    await client.models.list()
    return { ok: true }
  } catch (err) {
    if (err instanceof OpenAI.AuthenticationError) return { ok: false, error: 'invalid_key' }
    if (err instanceof OpenAI.PermissionDeniedError) return { ok: false, error: 'invalid_key' }
    if (err instanceof OpenAI.RateLimitError) return { ok: false, error: 'rate_limited' }
    if (err instanceof OpenAI.APIError) return { ok: true } // reached API, key OK
    return { ok: false, error: 'network_error' }
  }
}

export async function verifyGeminiKey(key: string): Promise<VerifyResult> {
  const ai = new GoogleGenAI({ apiKey: key })
  try {
    // Pager — just request the first page
    const pager = await ai.models.list()
    // If no exception, key is valid
    void pager
    return { ok: true }
  } catch (err) {
    const msg = (err as Error).message ?? ''
    if (msg.includes('API_KEY_INVALID') || msg.includes('PERMISSION_DENIED')) {
      return { ok: false, error: 'invalid_key' }
    }
    if (msg.includes('RESOURCE_EXHAUSTED')) return { ok: false, error: 'rate_limited' }
    // If it's a GoogleGenAIError it reached the API
    return { ok: false, error: 'network_error' }
  }
}
```

Note: `[ASSUMED]` — The exact Gemini error class name and error message patterns for `@google/genai` v2 SDK should be verified by reading the SDK source or testing against the API. The error handling pattern above is based on the AI-SPEC guidance and common SDK conventions.

### Pattern 6: DB Migration for Multi-Provider Keys

The current `creator_settings` table has one `encrypted_api_key` column. Phase 4 needs three provider-specific columns and an `active_provider` column.

### Pattern 7: compressHistory Adapter Refactoring

Currently `compressHistory()` in `anthropic.ts` takes an `Anthropic` client directly. With Phase 4, compression must use the active provider (D-03). Refactoring options:

**Option A (recommended):** Refactor `compressHistory()` to accept `AIProvider` and call `adapter.stream()` with a compression prompt. This is the cleanest pattern.

**Option B:** Keep `compressHistory()` Anthropic-only as a fallback, always use Anthropic for compression if a key exists, regardless of active provider. This violates D-03 (users with only OpenAI or Gemini keys cannot compress history).

Use Option A. The `/invoke` route already has the active adapter — pass it directly.

### Anti-Patterns to Avoid

- **Parsing OpenAI `function.arguments` before `finish_reason === "tool_calls"`**: `function.arguments` is a raw string fragment during streaming. Only `JSON.parse()` after `finish_reason`.
- **Using `role: "system"` in Gemini `contents`**: Gemini SDK throws or ignores it. Filter out system-role messages before building `contents`.
- **Not keying OpenAI tool accumulator by `tc.index`**: With `parallel_tool_calls` enabled (default), multiple deltas can interleave. Concatenating to one string produces invalid JSON.
- **Treating Gemini function calls as partial (like OpenAI)**: `chunk.functionCalls` returns complete objects. Do not accumulate as strings and re-parse.
- **Calling `await stream.finalMessage()` (OpenAI)**: This buffers the entire response and kills first-token latency.
- **Leaving `assemblePromptArray()` unchanged**: It returns `MessageParam[]` (Anthropic type) and embeds `cache_control` on content blocks — incompatible with other providers. The route must be refactored to use `ProviderMessage[]` instead, with `cache_control` moved inside `AnthropicAdapter`.
- **Importing from `@anthropic-ai/sdk` in the route (`ai.ts`)**: After refactoring, the route must not import Anthropic SDK types directly; only `AIProvider`, `ProviderMessage`, `ProviderTool` from `@panelito/types`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OpenAI streaming tool call accumulation | Custom binary parser for argument JSON | `tc.index` keyed accumulator + `JSON.parse()` after `finish_reason` | OpenAI parallel tool calls have index-based ordering; concatenating without index key breaks JSON |
| Gemini streaming function call extraction | Custom parser for `candidates[].content.parts` | `chunk.functionCalls` accessor | Google SDK provides a convenience accessor that handles the parts traversal and returns `FunctionCall[]` |
| Cross-provider API key encryption | New encryption library | Existing `encryptKey()`/`decryptKey()` from `apps/api/src/lib/crypto.ts` (AES-256-GCM) | Already battle-tested in Phase 1; same function works for any string key |
| Provider verification test calls | Custom HTTP call to provider APIs | Use provider SDK clients: `client.models.list()` (OpenAI), `ai.models.list()` (Gemini), existing `verifyApiKey()` pattern (Anthropic) | SDK clients handle auth headers, TLS, retry logic |
| `PanelWidgetSchema` validation | Provider-specific validation | Existing `PanelWidgetSchema.safeParse()` from `packages/types/src/panel.ts` | Schema is already correct for all providers — only the adapter output format differs |
| Session prompt assembly | Provider-specific prompt builders | Shared `ProviderMessage[]` assembled in route; each adapter converts internally | Single source of truth for context; adapters are pure converters, not assemblers |

**Key insight:** The adapter pattern means each adapter is a thin conversion layer. All business logic (context assembly, validation, cap counting, SSE writing) stays in the route unchanged. Adding a provider = adding one file under `adapters/`.

## Common Pitfalls

### Pitfall 1: assemblePromptArray returns Anthropic types

**What goes wrong:** The current `assemblePromptArray()` in `anthropic.ts` returns `MessageParam[]` and uses `cache_control: { type: 'ephemeral' }` on content blocks — an Anthropic SDK extension. If the route passes this array directly to `OpenAIAdapter.stream()` or `GeminiAdapter.stream()`, it passes objects containing `cache_control` metadata that OpenAI and Gemini SDKs will reject or ignore silently.

**Why it happens:** The function was written Anthropic-specific (correctly) in Phase 2. Phase 4 makes the route provider-agnostic.

**How to avoid:** Refactor `assemblePromptArray()` to return `ProviderMessage[]` (without `cache_control`). Move the `cache_control` injection into `AnthropicAdapter.stream()` when converting `ProviderMessage[]` to `MessageParam[]`. The adapter is the correct place for provider-specific extensions.

**Warning signs:** TypeScript error when passing `MessageParam[]` to `stream(messages: ProviderMessage[], ...)` after the interface refactor.

### Pitfall 2: Existing tests break on interface refactor

**What goes wrong:** `ai-provider.test.ts` imports `AnthropicAdapter` and `renderPanelTool` from `./ai-provider`. After the refactor moves `AnthropicAdapter` to `./adapters/anthropic`, imports break. The test also creates mock `MessageParam[]` for stream calls.

**Why it happens:** Test was written against the old interface before types moved to `packages/types/src/ai.ts`.

**How to avoid:** Update test imports first. Create `packages/types/src/ai.ts` with the new types. Update `ai-provider.ts` to export only the `AIProvider` interface. Move `AnthropicAdapter` to `adapters/anthropic.ts`. Update all import paths. Re-run tests after each step.

**Warning signs:** Test import errors; TypeScript "has no exported member" errors.

### Pitfall 3: DB migration breaks existing route references

**What goes wrong:** Renaming `encrypted_api_key` → `anthropic_api_key` in the DB migration (0006) makes all existing Supabase queries fail at runtime with "column does not exist" errors. The column is referenced in `keys.ts` (encrypt/upsert), `settings.ts` (select/update), and `ai.ts` (decrypt for provider key).

**Why it happens:** Column rename is not automatically reflected in TypeScript code.

**How to avoid:** The migration and code changes must be applied together. Plan the migration task to include updating all query strings referencing `encrypted_api_key`. Order: write migration → update all route code referencing that column → test.

**Warning signs:** Runtime Supabase errors `column "encrypted_api_key" does not exist` in logs.

### Pitfall 4: `done` event never yielded on adapter error

**What goes wrong:** If the adapter throws mid-stream (network error, API auth failure), the SSE connection hangs indefinitely — clients see a spinner with no resolution. The AI-SPEC defines a 30s timeout guardrail for this.

**Why it happens:** Async generators that throw instead of `return` do not yield a final event.

**How to avoid:** Wrap the entire `stream()` generator body in a `try/finally` block. Always `yield { type: 'done' }` in the `finally` block. The route's error handler catches the error AFTER done is already emitted.

**Warning signs:** SSE connections accumulate without closing in dev; clients frozen after AI call.

### Pitfall 5: `@google/genai` import is confused with `@google/generative-ai`

**What goes wrong:** Auto-import or copy-paste from old Google docs picks up `@google/generative-ai` (legacy v1 SDK) instead of `@google/genai` (v2 SDK). The v1 SDK has a different API: `new GoogleGenerativeAI(apiKey)` instead of `new GoogleGenAI({ apiKey })`, and `model.generateContentStream()` instead of `ai.models.generateContentStream()`.

**Why it happens:** The two packages coexist on npm. Documentation for v1 is more prevalent online. Package names differ by only one letter.

**How to avoid:** Only install `@google/genai`. Add a `// @google/genai v2 — do NOT use @google/generative-ai` comment in the import block of `adapters/gemini.ts`. TypeScript errors at compile time if v1 package is accidentally imported (different export names).

**Warning signs:** TypeScript error `Property 'models' does not exist on type 'GoogleGenerativeAI'`.

### Pitfall 6: OpenAI `gpt-4o` / `gpt-4o-mini` model IDs

**What goes wrong:** Using deprecated model IDs `gpt-4o` or `gpt-4o-mini` causes `404 model_not_found` API errors. These models were deprecated February 2026.

**Why it happens:** Training data and older documentation reference these model names.

**How to avoid:** Use `gpt-5.4` (analysis) and `gpt-5.4-mini` (compression/categorization) as documented in AI-SPEC.md §4. [ASSUMED — model names were validated in AI-SPEC.md generation, not re-verified in this research session against live OpenAI API]

**Warning signs:** `404 model_not_found` API error on first OpenAI adapter call.

### Pitfall 7: `active_provider` not read in session context

**What goes wrong:** The `/invoke` route currently hardcodes `new AnthropicAdapter(plaintextKey)`. After refactoring, the route needs to read `active_provider` from `creator_settings` before creating the adapter. If this read is missing, all sessions use Anthropic regardless of settings.

**Why it happens:** Easy to forget to add the DB read when the adapter factory is wired in.

**How to avoid:** The settings query in step 5 of the route (`creator_settings.select('encrypted_api_key')`) must be expanded to also select `anthropic_api_key`, `openai_api_key`, `gemini_api_key`, and `active_provider`. Then pass to `createAdapter(active_provider, resolvedKey)`.

### Pitfall 8: Supabase column-level security for new key columns

**What goes wrong:** New `openai_api_key` and `gemini_api_key` columns are readable by the `authenticated` PostgREST role by default (table-level SELECT was granted in migration 0002). If column-level REVOKE is not added for the new columns, a browser-side Supabase query could read them.

**Why it happens:** Supabase inherits table grants to new columns. The existing REVOKE in migration 0002 only covers `encrypted_api_key` (soon to be `anthropic_api_key`).

**How to avoid:** Migration 0006 must explicitly `REVOKE SELECT (openai_api_key, gemini_api_key, anthropic_api_key)` from `authenticated`. Same pattern as migration 0002.

// AnthropicAdapter converts ProviderMessage[] → MessageParam[] with cache_control

async *stream(
  messages: ProviderMessage[],
  tools: ProviderTool[],
  options: { model: string; maxTokens: number; system?: string }
): AsyncIterable<AIStreamEvent> {
  const client = new Anthropic({ apiKey: this.apiKey })

  // Convert ProviderTool[] → Anthropic.Tool[]
  const anthropicTools: Anthropic.Tool[] = tools.map((t) => ({
    name: t.name,
    description: t.description,
    input_schema: { type: 'object' as const, ...t.parameters },
  }))

  // Convert ProviderMessage[] → MessageParam[]
  // cache_control is Anthropic-specific — never in ProviderMessage
  const anthropicMessages: MessageParam[] = messages.map((m) => ({
    role: m.role as 'user' | 'assistant',
    content: m.content,
  }))

  const streamParams: Anthropic.MessageStreamParams = {
    model: options.model,
    max_tokens: options.maxTokens,
    tools: anthropicTools,
    messages: anthropicMessages,
    ...(options.system ? { system: options.system } : {}),
  }
  // ... rest of existing AnthropicAdapter stream logic unchanged
}

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | Yes (API key verification per provider) | Extend existing `verifyApiKey()` pattern; provider SDK auth error classes |
| V3 Session Management | No | Supabase session unchanged |
| V4 Access Control | Yes (column-level REVOKE for new key columns) | Migration 0006 must REVOKE `(openai_api_key, gemini_api_key, anthropic_api_key)` from `authenticated` |
| V5 Input Validation | Yes (provider enum, key format) | Zod — `ProviderSchema.parse()` before DB write; per-provider key format validation |
| V6 Cryptography | Yes (key storage) | Existing `encryptKey()`/`decryptKey()` (AES-256-GCM) — apply identically to all 3 key columns |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Unauthorized key read (new columns) | Information Disclosure | REVOKE SELECT on `openai_api_key`, `gemini_api_key`, `anthropic_api_key` from `authenticated` role in migration 0006 |
| Provider enumeration via error message | Information Disclosure | Return generic `invalid_key` error regardless of which provider produced it; do not leak provider-specific error details to browser |
| Rate limit abuse on verification endpoint | Denial of Service | Existing rate limiter (5/min/user in `keys.ts`) — apply same limit to multi-provider verify calls |
| Key for wrong provider stored in wrong column | Tampering | Per-provider format validation in `ApiKeyVerifyRequestSchema` (OpenAI keys start with `sk-`, Gemini keys with `AI`) before accepting |
| Active provider tampered in transit | Tampering | `active_provider` is updated via authenticated `PUT /api/settings` — RLS enforces `auth.uid() = user_id` |
