<interfaces>
Hono route (`apps/api/src/routes/messages.ts`):

- `POST /api/sessions/:id/messages` — requires auth (creator OR guest anonymous JWT). Body validated against `MessageInsertInputSchema` (content trimmed, 1-4000 chars; optional `display_name?: string` used only when the user is anonymous). Server (a) re-checks `session.status === 'active'`, (b) resolves `display_name`, (c) inserts via service-role client, (d) broadcasts the inserted row on `session:${id}` event `new_message` fire-and-forget. Returns 201 + row.
- `GET /api/sessions/:id/messages` — requires auth. Returns the last 200 messages for the session ordered by `created_at ASC`. Filters out other path_ids (Phase 1: only `path_id = 'main'`).

Web hook (`apps/web/hooks/use-session-channel.ts`):

- `useSessionChannel(sessionId: string, onMessage: (msg: Message) => void): void` — subscribes to `supabase.channel(session:${sessionId})` on mount, registers broadcast listener for `new_message`, calls `.subscribe()`. Cleanup removes the channel.

Web hook (`apps/web/hooks/use-typing-presence.ts`):

- `useTypingPresence(sessionId: string, userId: string, displayName: string): { typingUsers: TypingUser[]; setTyping: (b: boolean) => void }` — Presence channel `presence:${sessionId}` keyed by userId. `setTyping(true)` is throttled to <=1 call/sec.

Web hook (`apps/web/hooks/use-long-press.ts`):

- `useLongPress(handler, ms = 500): TouchAndMouseProps` — returns event handlers; on 500ms hold calls `navigator.vibrate?.(10)` and `handler()`.
- `useDoubleTap(handler, ms = 300): { onClick }` — two clicks within 300ms.

Zustand store (`apps/web/store/session-store.ts`):

- `useSessionStore` shape: `{ messages: Message[]; addMessage(msg); setMessages(msgs); typingUsers: TypingUser[]; setTypingUsers(users) }`. `addMessage` de-duplicates by `id`.
</interfaces>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (creator or guest) -> Hono `/messages` POST | Authenticated; payload size-capped at 4000 chars; server re-checks session.status; RLS gates the INSERT |
| browser -> Supabase Realtime broadcast | Direct from client; broadcast payload is the same row already persisted; no extra trust elevation |
| browser -> Supabase Presence | Each user can only track themselves (key = userId); display_name is user-supplied and rendered untrusted (XSS controls below) |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-05-01 | Tampering | Message content with HTML/script injection | mitigate | React JSX escapes interpolated text by default. The MessageBubble renders `{message.content}` as text node only — never `dangerouslySetInnerHTML`. Display name same treatment. |
| T-05-02 | Spoofing | Forged broadcast on a session channel (attacker subscribes + sends fake `new_message`) | mitigate | The MessageList ALWAYS fetches the persisted row by `id` on first render (initial history call); broadcasts are treated as cache-warming. Any broadcast for a message ID that does NOT exist in the database next-render-cycle could be flagged — accepted for Phase 1 since fake broadcasts would also fail RLS on subsequent reads. Hardening: Plan 06 considers verifying broadcasts by re-fetching from DB. |
| T-05-03 | Information Disclosure | A guest tampers with localStorage to use another guest's token | mitigate | Tokens are signed JWTs; tampering invalidates the signature; Supabase Auth rejects. RLS additionally requires `auth.uid() = author_id` on INSERT. |
