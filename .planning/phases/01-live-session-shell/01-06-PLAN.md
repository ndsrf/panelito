<interfaces>
Crypto module (`apps/api/src/lib/crypto.ts`):

- `encryptKey(plaintext: string, secret: string): string` — uses `crypto.createCipheriv('aes-256-gcm', Buffer.from(secret, 'hex'), iv)`. Returns base64 of `${iv}:${authTag}:${ciphertext}` (all base64-encoded, colon-separated for round-trip).
- `decryptKey(payload: string, secret: string): string` — reverses; throws on auth-tag mismatch.

Anthropic helpers (`apps/api/src/lib/anthropic.ts`):

- `verifyApiKey(key: string): Promise<{ ok: true } | { ok: false; error: 'invalid_key' | 'rate_limited' | 'network_error' }>` — instantiates `new Anthropic({ apiKey: key })` and calls `messages.create({ model: 'claude-sonnet-4-6', max_tokens: 1, messages: [{ role: 'user', content: 'ping' }] })`. On 401 -> `invalid_key`. On 429 -> `rate_limited`. On any network error -> `network_error`. On 200 -> `ok: true`.
- `assemblePromptArray(opts: { systemPrompt: string; personaInstructions: string; historicalSummary: string; recentMessages: Message[]; userMessage: string }): MessageParam[]` — produces the array with `cache_control: { type: 'ephemeral' }` at the END of the static prefix block (system + persona + summary). Dynamic tail (recentMessages + userMessage) is AFTER the breakpoint. Returns the array; does not invoke Claude. (Phase 2 wires the invocation.)

Hono routes (`apps/api/src/routes/keys.ts`):

- `POST /api/keys/verify` — requires auth. Body: `{ key: string }` (Zod: starts with `sk-ant-`, length >= 50). Calls `verifyApiKey(key)`. On `ok`: encrypts via `encryptKey(key, env.KEY_ENCRYPTION_SECRET)`, upserts into `creator_settings { user_id, encrypted_api_key, api_response_cap (default 150 on insert) }`, returns 200 + `{ success: true }`. On failure: returns 400 + `{ success: false, error }`.
- `GET /api/keys/status` — requires auth. Returns `{ has_api_key: boolean; last4: string | null }`. NEVER returns the key or the encrypted blob.
- `DELETE /api/keys` — requires auth. NULLs out `encrypted_api_key` (does not delete the row — preserves the cap).

Hono routes (`apps/api/src/routes/settings.ts`):

- `GET /api/settings` — requires auth. Returns `CreatorSettings` shape: `{ user_id, has_api_key, api_response_cap, updated_at }`. NEVER returns the encrypted key.
- `PUT /api/settings` — requires auth. Body: `{ api_response_cap?: number }` (Zod: int 1-10000 per the DB check). Updates only the fields present in the body.

Hono routes (`apps/api/src/routes/ai.ts`):

- `POST /api/sessions/:id/invoke` — requires auth + creator_id match. For Phase 1, this route assembles the prompt array via `assemblePromptArray(...)` and returns `c.json({ status: 'scaffolded', prompt_array_length: array.length, cache_breakpoint_position: idx }, 501)`. (Returns 501 Not Implemented because no actual Claude call fires yet — Phase 2 replaces with the streaming SSE handler. This route exists in Phase 1 only to lock the contract.)

Web helper (`apps/web/lib/creator-settings.ts`):

- `getCreatorSettings(): Promise<CreatorSettings>` — server-only; reads from the Hono /api/settings endpoint with the user's JWT.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> Hono `/api/keys/verify` | Plaintext key crosses this boundary over HTTPS in prod, HTTP locally; server immediately handshakes Anthropic and encrypts at rest |
| Hono -> api.anthropic.com | Direct API call with the candidate key; never logged |
| Postgres `creator_settings.encrypted_api_key` -> service-role decrypt -> in-memory plaintext -> Anthropic SDK -> discarded | The plaintext key lives only in memory for the duration of one request |
| browser -> Postgres (direct via Supabase client) | Column-level grant blocks `encrypted_api_key` from being read by `authenticated`; only `user_id`, `api_response_cap`, `updated_at` are readable |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | API key leakage via DB select | mitigate | Column-level REVOKE on `encrypted_api_key` from `authenticated` role (migration 0002); only service-role can read; service-role lives only in the Hono process; service-role key is in `.env`, never in client. |
| T-06-02 | Information Disclosure | API key leakage via Hono response body | mitigate | GET /api/keys/status returns ONLY `{ has_api_key, last4 }`. GET /api/settings returns `{ user_id, has_api_key, api_response_cap, updated_at }` — `encrypted_api_key` is NEVER in any response. Unit tests assert response shape. |
| T-06-03 | Information Disclosure | Key leakage via logs | mitigate | `anthropic.ts` `verifyApiKey` logs ONLY the error class name (`error.constructor.name`), never the key. Hono request logger (if added later) must redact `Authorization` header and request body for `/api/keys/*` routes. |
| T-06-04 | Tampering | At-rest tampering of `encrypted_api_key` (someone with DB write but not the secret) | mitigate | AES-256-GCM auth tag detects any byte change; `decryptKey` throws; the Hono route catches and treats as "no key" (forcing re-onboarding). |
| T-06-05 | Spoofing | An attacker pastes a valid Anthropic key that is not theirs (stolen credential) | accept | Out of platform scope; BYOK explicitly trusts the user to bring their own key. Anthropic's own abuse detection handles stolen keys. |
| T-06-06 | Elevation of Privilege | User A reads User B's settings | mitigate | All key/settings routes filter by `c.get('user').id`; RLS on `creator_settings` policy requires `auth.uid() = user_id`. |
