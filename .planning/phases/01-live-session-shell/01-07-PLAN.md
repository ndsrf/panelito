<interfaces>
Auto-freeze module (`apps/api/src/lib/auto-freeze.ts`):

- `startAutoFreezeTracker(supabase): void` — runs once at API boot. Subscribes to `presence:*` channels for all `active` sessions on startup AND on session-create events. Maintains an in-memory `Map<sessionId, { creatorId, lastSeenAt, timerHandle }>`.
- `onCreatorPresenceChange(sessionId, isOnline): void` — called by the presence-sync handler. If `isOnline = false`: starts a 30s + 15min nested timer. If `isOnline = true` during the 30s window: cancels both timers silently (SESS-11). At 30s+15min elapsed: calls `freezeSession(sessionId, reason: 'auto_freeze_creator_absent')` which sets `status='frozen'` and broadcasts `session_status_change` on `session:${id}`.

The 30s + 15min structure matches REQUIREMENTS.md SESS-11: "applies a 30-second grace buffer before treating an admin disconnect as a real absence; if the creator's device reconnects and emits presence within that window, the timer resets silently". So the implementation is: presence drop -> wait 30s of grace; if still gone -> mark "real absence" -> start 15min countdown -> auto-freeze on expiry. A reconnect at ANY point cancels everything and resets.

Auto-name module (`apps/api/src/lib/auto-name.ts`):

- `maybeAutoName(supabase, sessionId): Promise<void>` — called after each successful message INSERT. Conditions: session.title is NULL OR matches the placeholder regex; AND message count for this session reached EXACTLY 3 (not >3 — only the 3rd insert triggers). If both true: read the 3 messages, call the auto-name strategy. Phase 1 strategy: deterministic — derive a 2-3 word label from the most frequent non-stopword nouns/verbs in the 3 messages using a simple word-count heuristic, capitalized. (Phase 2 swaps the implementation for a flash Claude call per AI-09 wording in REQUIREMENTS.md.) On result: UPDATE sessions.title, broadcast `session_status_change` event with `{ title }` payload.

Cap-guard module (`apps/api/src/lib/cap-guard.ts`):

- `checkCap(supabase, sessionId): Promise<{ ok: true } | { ok: false; reason: 'cap_reached' }>` — reads sessions.ai_response_count + .ai_response_cap. If count >= cap, returns not-ok.
- `incrementCount(supabase, sessionId): Promise<{ count, cap, threshold_crossed: 'warning' | 'cap' | null }>` — atomic increment via SQL (`update sessions set ai_response_count = ai_response_count + 1 where id = $1 returning ai_response_count, ai_response_cap`). Determines threshold: if previous count < 0.9*cap and new count >= 0.9*cap -> 'warning'. If new count >= cap -> 'cap'. On 'warning': insert a system message "AI activity at 90% of session cap (X / Y). Y - X responses remaining." On 'cap': insert a system message "AI cap reached (Y / Y). Session is now frozen." then call `freezeSession(sessionId, reason: 'cap_reached')`.

Rate-limit middleware (`apps/api/src/lib/rate-limit.ts`):

- `rateLimit({ keyFn, limit, windowMs }): MiddlewareHandler` — in-memory token bucket. `keyFn(c) -> string` is the bucket key (e.g. `${c.get('user').id}:messages`). Returns 429 with `Retry-After` header on overflow.

Web hooks:

- `useCreatorPresence(sessionId, isCreator)` — if `isCreator`, calls `presence.track({ role: 'creator', last_seen: now })` on a 5s heartbeat. Untracks on unmount. (Guests do not publish creator presence; they only consume it via the existing typing-presence channel — both share `presence:${sessionId}`.)
- `useSessionStatus(sessionId, initialStatus)` — subscribes to `session:${id}` event `session_status_change`. On message, updates the session store. Returns the latest status.
