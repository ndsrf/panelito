<interfaces>
Hono routes to implement in `apps/api/src/routes/sessions.ts`:

- `POST   /api/sessions` — body: `SessionCreateInput`; requires auth; returns `Session`. Generates `short_code` via DB function. Sets `creator_id = c.get('user').id`.
- `GET    /api/sessions/:id` — requires auth OR a guest token; returns `Session` or 404. Used by the workspace page.
- `POST   /api/sessions/:id/freeze` — requires auth + `creator_id` match; sets `status = 'frozen'`; returns updated `Session`.
- `POST   /api/sessions/:id/close` — requires auth + `creator_id` match; sets `status = 'closed'`; returns updated `Session`.
- `GET    /api/sessions/by-code/:code` — public (no auth); returns `{ id, title, creator_display_name, status }` (limited fields for the join landing page).
- `POST   /api/sessions/by-code/:code/guests` — body: `{ display_name: string }`; calls `supabase.auth.signInAnonymously()` server-side, creates a row in `messages` with no rows (the guest is recognized by `author_id` once they post), returns `{ session_id, guest_user_id, access_token, refresh_token }`. The client persists this to localStorage and uses the token for subsequent realtime/inserts.

Hono middleware in `apps/api/src/middleware/auth.ts`:
- `requireAuth(c, next)` — reads `Authorization: Bearer ${token}`, calls `supabase.auth.getUser(token)` via the service-role client, sets `c.set('user', user)` on success, returns 401 on failure.

Client helper `apps/web/lib/api.ts`:
- `apiFetch<T>(path: string, init?: RequestInit): Promise<T>` — wraps `fetch(${NEXT_PUBLIC_API_URL}${path})`, attaches the current Supabase JWT as `Authorization: Bearer ${token}` (read from `createClient().auth.getSession()` on the browser, or from cookies via server client on the server), surfaces typed errors.

Guest-session module `apps/web/lib/guest-session.ts`:
- `saveGuestSession(shortCode: string, session: { guest_user_id: string; access_token: string; refresh_token: string; display_name: string }): void` — `localStorage.setItem(\`panelito:guest:${shortCode}\`, JSON.stringify(session))`.
- `loadGuestSession(shortCode: string): GuestSession | null` — reads + parses.
- `clearGuestSession(shortCode: string): void` — removes the key.
- Caller MUST `await` `saveGuestSession` (synchronous, but the join action awaits it before `router.push` to prevent the SSR pitfall noted in RESEARCH.md).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (creator) → Hono `/api/sessions` | Authenticated; JWT bearer; server enforces `creator_id === user.id` on mutations |
| browser (guest, anonymous) → Hono `/api/sessions/by-code/:code/guests` | Public; rate-limited; payload size-capped |
| browser → Hono `GET /api/sessions/by-code/:code` | Public; returns a strictly-limited subset of session fields |
| browser → localStorage | Untrusted; localStorage values are session-scoped tokens that the server validates via Supabase Auth on every request |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | Guest claims another user's `author_id` | mitigate | Anonymous Supabase tokens have a unique `sub` claim; RLS on `messages` INSERT requires `auth.uid() = author_id`; a guest cannot insert with another guest's ID. |
| T-03-02 | Elevation of Privilege | Non-creator freezes or closes someone else's session | mitigate | `POST /freeze` and `POST /close` re-check `creator_id === c.get('user').id` in addition to RLS; on mismatch return 403. |
| T-03-03 | Information Disclosure | `GET /by-code/:code` leaks creator's full profile or AI cap | mitigate | Route explicitly selects `id, title, status` and joins for `creator_display_name` only; does NOT return `creator_id`, `ai_response_count`, `ai_response_cap`, or any settings fields. |
| T-03-04 | Tampering | Guest tampers with localStorage to claim a different `guest_user_id` | mitigate | The `access_token` in localStorage is a signed Supabase JWT; tampering invalidates the signature and Supabase Auth rejects it on the next API call. Display name tampering is moot — it appears on the user's own bubbles only. |
| T-03-05 | Denial of Service | Mass session creation by a logged-in user | mitigate | Add per-user rate limit on `POST /api/sessions` (max 10 / minute) via Hono middleware using an in-memory token bucket keyed by `user.id`. (Phase 1 ships with in-memory; Redis-backed is post-MVP.) |
| T-03-06 | Denial of Service | Mass guest signups against `POST /by-code/:code/guests` | mitigate | Per-IP rate limit (max 20 / minute) on the guests endpoint via the same middleware; 429 on overflow. |
| T-03-07 | Information Disclosure | Service-role key leakage in Hono error responses | mitigate | All routes wrap Supabase errors in a custom `toClientError(err)` helper that returns `{ error: 'internal' }` for unexpected errors (no raw error message bubbled to client). Detailed errors go to stderr only. |
| T-03-08 | Tampering | Short code collision (two sessions with same code) | mitigate | DB unique constraint on `sessions.short_code` plus the generator function regenerates on conflict (the migration's `generate_short_code()` retries up to 5 times). |
| T-03-09 | Information Disclosure | QR/join URL guessable | accept | 6-char Crockford base32 = ~1.07B possible codes; collision/brute force is impractical for the demo scale. Reassessed in production with longer codes if needed. |
| T-03-SC | Tampering | `qrcode.react` package | mitigate | Verified Approved in Plan 01's audit (github.com/zpao/qrcode.react, npm-popular, 4.x). Pinned to `4.2.0`. |
